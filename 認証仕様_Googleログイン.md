# 清田自動車 工程管理アプリ 認証仕様（Googleログイン）

---

## 1. 概要

- **目的**  
  - 本ドキュメントは、清田自動車 工程管理アプリにおける **Googleアカウントによるログイン機能** と、**指定された Google アカウントのみが利用できる制限機能** の仕様を定義する。  
  - 既存のログイン画面（現在は UI モック）を実運用可能な認証機構へ拡張するための指針とする。

- **対象**  
  - 本アプリのフロントエンド（React + Vite）  
  - 認証基盤として利用する **Firebase Authentication（Google プロバイダ）**  
  - 必要に応じてバックエンド/API（将来的な拡張を想定）

---

## 2. 認証・認可の方針

- **認証方式**  
  - Google OAuth 2.0 を利用した **Firebase Authentication / Google サインイン** を採用する。
  - フロントエンドは Firebase の JavaScript SDK を用いてサインイン処理を行う。

- **アカウント制限（認可）**  
  - ログイン可能なユーザーは、あらかじめ定義した **許可メールアドレス一覧（Allow List）** に含まれる Google アカウントのみとする。
  - 許可リストに含まれないアカウントでログインを試みた場合、即座にサインアウトし、エラーメッセージを表示する。
  - 将来的には、  
    - 「メールアドレス単位の許可リスト」  
    - 「ドメイン単位の制限（例: `@example.co.jp` のみ）」  
    を切り替え可能な設計を想定する。

- **セッション管理**  
  - Firebase Authentication が発行する ID トークンを利用し、フロントエンドでは「ログイン済みかどうか」を判定する。  
  - 将来的にバックエンド/API を追加する場合は、API へのリクエスト時に ID トークンをヘッダー送信し、サーバー側でも認証・許可リストチェックを行う。

---

## 3. 画面仕様

### 3.1 ログイン画面

- **表示タイミング**
  - 未ログイン状態でアプリにアクセスした場合、最初にログイン画面を表示する。
  - すでにログイン済みの場合は、ログイン画面をスキップしてメイン画面（カンバンボード）へリダイレクトする。

- **UI 要素**
  - アプリ名／ロゴ
  - 「Googleでログイン」ボタン（Firebase Authentication と連携）
  - 許可されていないアカウントや認証エラー時のメッセージ表示エリア
    - 例: 「このアカウントではログインできません。管理者にお問い合わせください。」

- **状態表示**
  - ログイン処理中はロードインジケータ（スピナーなど）を表示し、ボタンを連打できないようにする。

### 3.2 メイン画面（カンバンボード／ガントビュー）

- ログイン済みユーザーのみが閲覧・操作できる。
- ヘッダー右上のユーザーアイコン／メールアドレス表示を行う。
- ドロップダウンメニューから「ログアウト」を実行可能とする。

---

## 4. 機能要件

### 4.1 ログイン処理

- **前提条件**
  - Firebase プロジェクトが作成済みで、  
    - Authentication > Sign-in method にて「Google」が有効化されていること。
    - Web アプリ用の API キー・projectId などが取得済みであること。

- **処理フロー（フロントエンド）**
  1. ユーザーが「Googleでログイン」ボタンをクリック。
  2. Firebase Authentication の `signInWithPopup` または `signInWithRedirect` を使用して Google サインイン画面を表示。
  3. Google 側で認証成功後、Firebase からユーザー情報（メールアドレス、表示名、UID など）を取得。
  4. 取得したメールアドレスが **許可メールアドレス一覧** に含まれているか判定。
      - **許可されている場合**  
        - ログイン状態としてアプリケーション state に保存。  
        - メイン画面（カンバンボード）へ遷移。
      - **許可されていない場合**  
        - すぐに Firebase からサインアウト。  
        - ログイン画面にとどまり、エラーメッセージを表示。

### 4.2 許可メールアドレス一覧の管理

- **バージョン1（簡易実装・フロントエンドのみ）**
  - 許可リストをフロントエンド側の定数として定義する。  
    - 例: `ALLOWED_EMAILS = ['xxx@example.com', 'yyy@example.com']`  
  - ビルド時にリストが固定されるため、変更には再デプロイが必要。
  - 少人数運用を想定した暫定仕様。

- **バージョン2（将来拡張案）**
  - 許可リストを以下のいずれかで管理する:
    - バックエンドAPI（例: Node.js + DB）  
    - Firebase Remote Config  
    - Firestore コレクション（`allowedUsers` など）
  - ログイン後、バックエンドまたは Firestore 側でメールアドレスが許可されているかを再チェックし、NGならセッション無効化。

### 4.3 ログアウト処理

- ヘッダーのユーザーメニューから「ログアウト」を選択可能とする。
- 処理内容:
  - Firebase Authentication からサインアウトする。
  - アプリ内のユーザー state（React state / context）を初期化する。
  - ログイン画面へリダイレクトする。

## 5. 画面遷移・ルーティング

### 5.1 ルーティング構成（例）

- `/login` : ログイン画面
- `/` または `/board` : カンバンボード画面
- `/gantt` : 代車ガントチャート画面

### 5.2 認証ガード

- ルーティングレベルで **認証ガード** を実装する。
  - 未ログイン状態で `/` や `/gantt` にアクセスした場合、`/login` にリダイレクトする。
  - ログイン済み状態で `/login` にアクセスした場合、`/` にリダイレクトする。

---

## 6. データ項目・状態管理

### 6.1 ユーザー情報（フロントエンド）

- 少なくとも以下の情報をアプリケーション state で保持する。
  - `uid`: Firebase ユーザーID
  - `email`: メールアドレス
  - `displayName`: 表示名（任意）
  - `photoURL`: アイコン画像URL（任意）
  - `isAllowed`: 許可リストに含まれているかどうかのフラグ

- 状態管理方式の例
  - React Context (`AuthContext`) で全体に配布する。
  - または状態管理ライブラリ（Redux 等）を利用する。

### 6.2 永続化

- Firebase Authentication は標準でブラウザのストレージ（LocalStorage / IndexedDB 等）にセッションを保持する。
- 追加のユーザー情報やアプリ独自設定を保存する場合は、以下のいずれかを利用する。
  - `localStorage`
  - `sessionStorage`
  - Firestore / 自前バックエンド

## 7. セキュリティ要件

- **許可リストの秘匿性**
  - バージョン1（フロントエンドのみ）の場合、ビルド後のコードからメールアドレス一覧が見えてしまう点を理解したうえで、  
    - 利用ユーザーが限定されている  
    - 内部利用のみ  
    など、運用上許容できる前提で使用する。
  - より厳格なセキュリティが必要になった段階で、バックエンドまたは Firestore による管理に移行する。

- **ID トークンの利用**
  - 将来的にバックエンドを導入する際は、必ず Firebase の ID トークン検証をサーバー側で行う。
  - クライアントから API へは `Authorization: Bearer <ID_TOKEN>` 形式で送信する。

- **CSRF / XSS**
  - 本仕様では主に認証まわりの要件に限定するが、フォーム入力や外部リソース取り扱いにおける一般的な XSS 対策は別途考慮する。

---

## 8. エラーハンドリング仕様

- **Google 認証ポップアップがブロックされた場合**
  - ブラウザのポップアップブロックによりサインインが失敗する可能性があるため、  
    - 「ポップアップがブロックされました。ブラウザの設定を確認してください。」といったガイドメッセージを表示する。

- **ネットワークエラー**
  - ネットワーク不通時には、再試行ボタンや「時間をおいて再度お試しください」などのメッセージを表示。

- **許可されていないアカウント**
  - メッセージ例:  
    - 「このGoogleアカウントではログインできません。」  
    - 「利用を希望される場合は、管理者へアカウント登録を依頼してください。」

- **その他 Firebase エラーコード**
  - 必要に応じて `auth/popup-closed-by-user` など主要なエラーコードに対して個別メッセージを用意する。

## 9. テスト観点

- **正常系**
  - 許可リストに含まれるメールアドレスでログインし、カンバンボードが表示されること。
  - ログイン状態でページリロードしても、再ログインなしで利用できること。

- **異常系**
  - 許可されていないメールアドレスでログインを試みると、エラーメッセージが表示され、メイン画面へ遷移しないこと。
  - ログアウト後、保護された画面に直接アクセスしてもログイン画面へリダイレクトされること。
  - ネットワーク断時やポップアップブロック時に、ユーザーが状況を理解できるメッセージが表示されること。

---

## 10. 今後の拡張案

- 許可リストを Firestore またはバックエンドで管理し、管理画面から「ユーザー追加／削除」ができるようにする。
- ロール（権限）管理の導入  
  - 例: 管理者、一般ユーザー、閲覧のみ などの役割を追加し、  
    - 一部機能（例: カード削除、設定変更）を管理者のみに制限する。
- ログイン履歴の記録  
  - いつ、どのアカウントがログインしたかを記録し、監査に利用できるようにする。
