## 0. 【テスト用】すぐにフロントに触ってもらう手順

### 0-A. とりあえず1回だけ公開する（約5分）

1. **ビルドする**
   ```powershell
   cd "プロジェクトのフォルダパス"
   npm ci
   npm run build
   ```
2. **Netlify Drop で公開する**
   - ブラウザで [https://app.netlify.com/drop](https://app.netlify.com/drop) を開く（Netlify アカウントがなければ無料で作成）。
   - 表示されたエリアに、プロジェクト内の **`dist` フォルダ** をドラッグ＆ドロップする。
   - 表示されたURLをフロント担当者に共有する。

**注意:** コードを変えたら、再度 `npm run build` してから `dist` をドラッグし直す必要があります。

---

### 0-B. すぐに内容を反映させたい場合（push で自動デプロイ）

「コードを push するだけで、テスト用URLにすぐ反映される」ようにしたい場合は、次のいずれかを選んで一度だけ設定します。設定後は **push するたびに自動でビルド・公開**されます。

| 方法 | 特徴 | 手順 |
|------|------|------|
| **GitHub + Netlify** | 無料・設定が簡単・URL例: `xxx.netlify.app` | [3.2 オプションB: Netlify](#32-オプションb-netlify) の「Git 連携で自動デプロイ」 |
| **GitHub + Vercel** | 無料・設定が簡単・URL例: `xxx.vercel.app` | [3.3 オプションC: Vercel](#33-オプションc-vercel) でリポジトリを連携 |
| **GitHub Pages** | 無料・URL例: `ユーザー名.github.io/リポジトリ名/` | [3.4 オプションD: GitHub Pages](#34-オプションd-github-pages) |
| **GitHub + Xserver** | 既存の Xserver のドメインで公開したい場合 | [3.5 オプションE: Xserver](#35-オプションe-xserver) の「GitHub Actions で push 時に自動デプロイ」 |

**おすすめ:** テスト公開だけなら **GitHub + Netlify** または **GitHub + Vercel** が最も手軽です。すでに Xserver でドメインを持っている場合は **GitHub + Xserver** で push のたびに FTP アップロードする方法があります。

## 1. 前提条件と準備

### 1.1 必要な環境

| 項目 | 内容 |
|------|------|
| Node.js | **v18 以上**（推奨: v20 LTS） |
| npm | v9 以上（Node.js に同梱） |
| エディタ / ターミナル | コマンドが実行できる環境 |

**Node.js のバージョン確認**

```powershell
node -v
npm -v
```

`v18.0.0` や `v20.x.x` のように表示されれば問題ありません。

### 1.2 ソースコードの取得

- 本番用のコードは **Git で管理**している場合、本番サーバーや CI で `git clone` または `git pull` で最新版を取得します。
- 手元のフォルダをそのままコピーする場合は、**`node_modules` フォルダは含めず**、プロジェクト直下のソースのみを渡します。

### 1.3 依存パッケージのインストール

プロジェクトのフォルダ（`package.json` がある場所）で、以下を実行します。

```powershell
cd "c:\Users\ken\新しいフォルダー\清田自動車\工程管理"
npm ci
```

- **`npm ci`** … `package-lock.json` に合わせて厳密にインストール。本番・CI ではこちらを推奨。
- 開発時と同様にインストールする場合は `npm install` でも構いません。

インストールが完了すると、`node_modules` フォルダが作成されます。

## 3. デプロイ先別の手順

本アプリは **静的サイト**（HTML + JS + CSS）のため、以下のいずれかの方法で公開できます。

---

### 3.1 オプションA: 自社サーバー（IIS / Apache / Nginx）

#### 手順概要

1. 上記「2. 本番ビルドの実行」で `dist` フォルダを作成する。
2. **`dist` フォルダの中身全体**を、Web サーバーの公開ディレクトリにコピーする。
3. SPA 用に「すべてのパスで `index.html` を返す」設定を行う。

#### IIS（Windows サーバー）の場合

1. **公開フォルダの指定**
   - IIS の「サイト」→ 対象サイト → 「基本設定」で、物理パスを `dist` の中身をコピーしたフォルダに設定（例: `C:\inetpub\wwwroot\kotei`）。

2. **URL 書き換えのインストール**
   - 「URL Rewrite」モジュールが入っていない場合は、[Microsoft の公式ページ](https://www.iis.net/downloads/microsoft/url-rewrite)からインストール。

3. **`web.config` を配置**
   - 公開フォルダ（`index.html` と同じ場所）に、次の内容で **`web.config`** を作成する。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <rule name="SPA Fallback" stopProcessing="true">
          <match url=".*" />
          <conditions logicalGrouping="MatchAll">
            <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
            <add input="{REQUEST_FILENAME}" matchType="IsDirectory" negate="true" />
          </conditions>
          <action type="Rewrite" url="/index.html" />
        </rule>
      </rules>
    </rewrite>
    <staticContent>
      <mimeMap fileExtension=".json" mimeType="application/json" />
    </staticContent>
  </system.webServer>
</configuration>
```

4. サイトを再起動し、ブラウザで `https://あなたのドメイン/` にアクセスして表示を確認する。

#### サブフォルダに配置する場合（例: `https://example.com/kotei/`）

- **Vite の設定変更**が必要です。`vite.config.js` に `base` を追加します。

```javascript
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  base: '/kotei/',   // サブパスに合わせて変更
})
```

- 変更後に **再度 `npm run build`** し、新しい `dist` をサーバーに配置します。

### 3.2 オプションB: Netlify

1. [Netlify](https://www.netlify.com/) にアカウント作成・ログイン。
2. **「Add new site」→「Deploy manually」** を選択。
3. **「dist」フォルダをドラッグ＆ドロップ**してアップロード。
4. デプロイ完了後、表示された URL で動作確認。

**Git 連携で自動デプロイする場合**

1. GitHub 等にリポジトリを push。
2. Netlify で「Import from Git」→ リポジトリを選択。
3. ビルド設定を次のように入力：
   - **Build command:** `npm run build`
   - **Publish directory:** `dist`
4. デプロイを実行。以降、`main` など指定ブランチに push すると自動で本番が更新されます。

**SPA 用リダイレクト設定（Netlify）**

プロジェクト直下に **`public/netlify.toml`** またはルートに **`netlify.toml`** を作成し、次を記載します。

```toml
[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

---

### 3.3 オプションC: Vercel

1. [Vercel](https://vercel.com/) にアカウント作成・ログイン。
2. **「Add New」→「Project」** で、Git リポジトリをインポートするか、**「Upload」** で `dist` フォルダをアップロード。
3. フレームワークは **Vite** を選択（自動検出される場合あり）。
4. **Root Directory** がプロジェクトルートであることを確認。
5. デプロイ後、表示された URL で確認。

Vercel は Vite の SPA を標準でサポートしているため、多くの場合そのままで動作します。サブディレクトリに置く場合は、Vite の `base` を設定してください。

---

### 3.4 オプションD: GitHub Pages

1. **`vite.config.js`** で、GitHub リポジトリ名を base に設定します。
   - 例: リポジトリが `https://github.com/ユーザー名/kiyota-kotei` の場合  
     `base: '/kiyota-kotei/'`

```javascript
export default defineConfig({
  plugins: [react()],
  base: '/kiyota-kotei/',  // リポジトリ名に合わせる
})
```

2. `npm run build` で `dist` を生成。
3. GitHub のリポジトリの **Settings → Pages** で、Source を **GitHub Actions** に設定。
4. リポジトリに次のワークフローを追加します。  
   ファイル名: **`.github/workflows/deploy.yml`**

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [ main ]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist
  deploy:
    needs: build-and-deploy
    runs-on: ubuntu-latest
    environment: github-pages
    steps:
      - uses: actions/deploy-pages@v4
```

5. `main` ブランチに push すると、自動でビルド・デプロイされ、  
   `https://ユーザー名.github.io/kiyota-kotei/` で公開されます。

### 3.5 オプションE: Xserver

Xserver（エックスサーバー）の契約済みドメインで公開する場合の手順です。**push するたびに自動で反映**させるには、下記「GitHub Actions で push 時に自動デプロイ」を設定してください。

#### 事前に用意するもの

- Xserver の FTP アカウント情報（サーバーパネル → **FTPアカウント設定** で確認）
  - **FTP サーバー（ホスト名）** … 例: `ftp.あなたのドメイン.com` または Xserver から案内されたホスト名
  - **FTP ユーザー名** … FTP 用のユーザーID
  - **FTP パスワード** … FTP 用パスワード（サーバーパスワードと同じ場合が多いです）
- アップロード先 … 通常は **`public_html`** 直下、またはサブフォルダ（例: `public_html/kotei`）

#### 手動でアップロードする場合

1. 手元で `npm run build` を実行し、**`dist` フォルダの中身**を用意する。
2. FTP クライアント（FileZilla、WinSCP など）で Xserver に接続する。
3. **public_html**（または公開したいサブフォルダ）に、`dist` の中身（`index.html` と `assets` フォルダなど）をそのままアップロードする。
4. サブフォルダに置く場合（例: `https://あなたのドメイン.com/kotei/`）は、**`vite.config.js`** で `base: '/kotei/'` を設定してからビルドし直す。
5. SPA 用に **`.htaccess`** を配置する（下記「SPA 用 .htaccess」を参照）。

#### GitHub Actions で push 時に自動デプロイする

**main ブランチに push するたびに、自動でビルドして Xserver の FTP にアップロード**されます。

**ステップ1: ワークフローファイルをリポジトリに含める**

プロジェクトに **`.github/workflows/deploy-xserver.yml`** を用意します（本手順書の「Xserver 用ワークフロー例」をコピーして作成）。リポジトリに push しておきます。

**ステップ2: GitHub の Secrets に FTP 情報を登録**

1. GitHub で対象リポジトリを開く → **Settings** → **Secrets and variables** → **Actions**
2. **New repository secret** で、次の3つを追加する。

   | Secret 名 | 値の例 | 説明 |
   |------------|--------|------|
   | `FTP_SERVER` | `ftp.あなたのドメイン.com` | FTP のホスト名（接続先） |
   | `FTP_USERNAME` | `あなたのFTPユーザー名` | FTP ユーザーID |
   | `FTP_PASSWORD` | `FTPのパスワード` | FTP パスワード |

3. **アップロード先フォルダ**用の Secret を追加する。

   | Secret 名 | 値の例 | 説明 |
   |------------|--------|------|
   | `FTP_REMOTE_DIR` | `public_html` | 公開用フォルダ。ルートで公開する場合は `public_html`。サブフォルダの場合は `public_html/kotei` など。 |

**ステップ3: 初回のみ手動で .htaccess を配置**

Xserver の **public_html**（または `FTP_REMOTE_DIR` で指定したフォルダ）に、次の内容で **`.htaccess`** を1つ置いておきます。GitHub Actions は `dist` の中身だけを上書きするため、`.htaccess` は別途 FTP で1回アップロードするか、`dist` に含めてビルド前に `public/.htaccess` に置いておく方法があります（後述）。

**ステップ4: 動作確認**

- `main` ブランチに push する。
- GitHub の **Actions** タブでワークフローが成功しているか確認する。
- ブラウザで Xserver の URL にアクセスし、アプリが表示されるか確認する。

**Xserver 用ワークフロー**

リポジトリには **`.github/workflows/deploy-xserver.yml`** が含まれています。上記のとおり GitHub の Secrets に `FTP_SERVER` / `FTP_USERNAME` / `FTP_PASSWORD` / `FTP_REMOTE_DIR` を設定すれば、`main` に push するたびに自動でビルド・アップロードされます。

- ルートで公開する場合: `FTP_REMOTE_DIR` に **`public_html`** を設定。
- サブフォルダ（例: `https://あなたのドメイン.com/kotei/`）で公開する場合: **`vite.config.js`** に `base: '/kotei/'` を設定し、`FTP_REMOTE_DIR` に **`public_html/kotei`** を設定してください。

**SPA 用 .htaccess（Xserver は Apache）**

アップロード先のフォルダに **`.htaccess`** を置き、存在しないパスは `index.html` に振り向けます。  
プロジェクトの **`public`** フォルダに `.htaccess` を置いておくと、`npm run build` 時に **`dist`** にコピーされます。

```apache
RewriteEngine On
RewriteBase /
RewriteRule ^index\.html$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.html [L]
```

- サブフォルダ（例: `/kotei/`）に配置した場合は、`RewriteBase /kotei/` と `RewriteRule . /kotei/index.html [L]` に変更してください。

## 5. よくあるトラブルと対処

### ビルドが失敗する

- **エラーメッセージを確認**し、不足しているパッケージや構文エラーを修正する。
- `node_modules` を削除してから `npm ci` をやり直す。

```powershell
Remove-Item -Recurse -Force node_modules
npm ci
npm run build
```

### 本番で「真っ白な画面」になる

- **JavaScript のパス**が誤っている場合があります。
  - サブフォルダに配置しているのに `base` を設定していない → `vite.config.js` の `base` を設定し、再ビルド。
- ブラウザの開発者ツール（F12）の **Console** にエラーが出ていないか確認する。

### リロードや直接 URL で 404 になる

- SPA のため、**すべてのパスで `index.html` を返す**設定が必要です。
  - IIS: 上記の `web.config` を配置。
  - Netlify: `netlify.toml` の redirect を設定。
  - Nginx: `try_files $uri $uri/ /index.html;` を設定。

### キャッシュで古い画面が出る

- 本番では Vite がファイル名にハッシュを付けるため、通常は自動で新しいファイルが読まれます。
- それでも古い場合、ブラウザの **スーパーリロード**（Ctrl+Shift+R または Ctrl+F5）を試すか、サーバー側でキャッシュヘッダを短くする設定を検討してください。

## 6. Googleスプレッドシート連携（カード作成時の自動入力）

アプリでカードを作成したときに、同じ内容を Google スプレッドシートのシートへ自動で1行追加する機能です。

### 6.1 必要なもの

- Google アカウント
- カード記録用の Google スプレッドシート（新規で作成可）
- 本プロジェクト内の **`gas/kotei_sheet_sync.gs`** を Google Apps Script（GAS）にコピーしてデプロイする

### 6.2 スプレッドシートの準備

1. [Google スプレッドシート](https://sheets.google.com/) で新規スプレッドシートを作成する。
2. 1枚目のシート名を **「カード履歴」** にする（または GAS 内の `SHEET_NAME` を同じ名前にする）。
3. 1行目に次のヘッダーを入力する（左のセルから順に）。

   | A | B | C | D | E | F | G | H | I | J | K | L | M | N | O |
   |---||---||---||---||---||---||---||---||---||---||---||---|---|---|
   | 作成日時 | カードID | ステータス | メーカー | モデル | 車番 | 顧客名 | 入庫日 | 納車日 | 代車種別 | 受付担当 | 鈑金担当 | 塗装担当 | カード色 | 説明 |

### 6.3 Google Apps Script の設定とデプロイ

1. スプレッドシートのメニュー **「拡張機能」→「Apps Script」** を開く。
2. 既定の `Code.gs` の内容を削除し、**`gas/kotei_sheet_sync.gs`** の内容をすべて貼り付けて保存する。
3. **`kotei_sheet_sync.gs` の先頭にある設定**を編集する。
   - **`SPREADSHEET_ID`** … スプレッドシートの URL の `/d/` と `/edit` の間の文字列（例: `1abc...xyz`）。
   - **`SHEET_NAME`** … データを書き込むシート名（例: `カード履歴`）。
4. **「デプロイ」→「新しいデプロイ」** をクリック。
   - 種類で **「ウェブアプリ」** を選択。
   - 「次のユーザーとして実行」: **自分**、「アクセスできるユーザー」: **全員** にして「デプロイ」。
   - 表示された **「ウェブアプリの URL」** をコピーする（例: `https://script.google.com/macros/s/xxxxx/exec`）。

### 6.4 アプリ側の設定

1. プロジェクト直下に **`.env`** ファイルを作成する（Git にコミットしない場合は `.gitignore` に `.env` を追加）。
2. 次の1行を記述する（URL は 6.3 でコピーしたものに置き換える）。

   ```
   VITE_SHEET_SYNC_URL=https://script.google.com/macros/s/あなたのID/exec
   ```

3. **`.env` を変更したあとは `npm run build` をやり直す**必要があります（Vite はビルド時に環境変数を埋め込みます）。

これで、アプリで「カード作成」を実行すると、同じ内容がスプレッドシートの「カード履歴」シートに1行ずつ追記されます。スプレッドシート連携を一時的に止めたい場合は、`.env` の `VITE_SHEET_SYNC_URL` を空にするか削除してから再ビルドしてください。

## まとめ

1. **Node.js を入れ、`npm ci` → `npm run build`** で `dist` を作成する。  
2. **デプロイ先**（自社サーバー / Netlify / Vercel / GitHub Pages）に合わせて、`dist` の配置と SPA 用の設定（書き換え・リダイレクト）を行う。  
3. **リリース前チェックリスト**で表示・動作を確認する。  
4. **Google スプレッドシート連携**を使う場合は、GAS をデプロイし、`.env` に `VITE_SHEET_SYNC_URL` を設定してからビルドする。

不明点や環境に応じた細かい設定が必要な場合は、この手順書をベースに社内の担当者と相談してください。
